# ADR-007: stay 삭제는 물리 삭제가 아니라 `deleted` 컬럼 소프트 삭제로 한다

- **상태**: 확정 (2026-07-17)
- **결정**: 사용자가 stay를 지울 때 행을 `DELETE`하지 않고 `stays.deleted`(0/1) 플래그만 세운다. 모든 조회는 `WHERE deleted = 0`으로 거른다.
- **대가**: 삭제 행이 물리적으로 남아 DB가 미세하게 커진다 — 개인앱 규모에서 무시 가능.

## 배경 — 왜 그냥 지우면 안 되나

체류판정(`detectStays`)은 전체 재계산이 아니라 **마지막 처리 지점 이후의 신규 points만 증분 처리**한다. 증분 커서는 이미 확정된 stays의 최신 지점에서 이어진다.

여기서 최신 stay를 물리 삭제하면, 커서가 그 stay가 덮던 구간만큼 **뒤로 밀린다.** 다음 수집·판정이 돌 때 같은 points가 다시 증분 대상이 되고, 방금 지운 stay가 **그대로 재생성**된다. 사용자 입장에선 "지웠는데 다시 생김"이다.

즉 삭제는 단순한 행 제거가 아니라 **증분 파이프라인의 상태와 얽힌 결정**이다.

## 검토한 대안

| 대안 | 장점 | 단점 | 판단 |
|---|---|---|---|
| ① 물리 삭제 (`DELETE FROM stays`) | 스키마 단순, DB 안 커짐 | 증분 커서가 밀려 지운 최신 stay가 부활. 재-import 시 삭제·라벨 이력도 소실 | 부활 문제로 탈락 |
| **② 소프트 삭제 (채택)** | 행이 남아 커서 불변 → 부활 없음. 재-import 시 `(start_ts, source)` 유니크 + `INSERT OR IGNORE`로 삭제·라벨 상태 보존 | 삭제 행이 물리적으로 남음 | 개인앱 규모에서 무시 가능 |

## 결과

**얻는 것**

- 증분 커서가 삭제와 무관하게 유지된다 — 지운 stay가 부활하지 않는다.
- 재-import가 라벨·삭제 상태를 보존한다 — 같은 데이터를 다시 넣어도 사용자의 정리 결과가 살아 있다.

**잃는 것 / 지켜야 할 것**

- **모든 조회 경로가 `deleted = 0`을 걸어야 한다** — `getStaysByRange`(통계·목록)·`getDatesWithData`(달력 점)·`getLabeledStays`(라벨 근접 조회) 전부 필터 적용. 새 조회를 추가할 때 이 필터를 빠뜨리면 지운 stay가 화면·통계·라벨 상속에 되살아난다.
- 삭제 행이 물리적으로 남는다 — 진짜 청소가 필요해지면 오래된 `deleted = 1` 행을 정리하는 잡을 후속으로 둔다. 지금 규모에선 불필요.
