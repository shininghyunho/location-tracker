# ADR-006: 모든 시각을 +09:00 고정 오프셋으로 저장하고, 날짜 경계는 문자열 사전순으로 자른다

- **상태**: 확정 (2026-07-18 명문화 — 구현 초기부터 암묵 적용)
- **결정**: 저장되는 모든 `ts`(points·stays)는 **`+09:00` 고정 오프셋 ISO 문자열**로 통일한다. 날짜·기간 경계 자르기는 그 문자열의 **사전순 비교**로, 시간 길이·간격 계산은 **`Date.parse`의 절대 epoch**로 한다. 앱 전체가 단일 타임존(KST)을 가정한다.
- **대가**: 해외 이동 시 현지 자정이 아니라 KST 자정으로 하루가 잘린다 — 단일 사용자·국내 전용이라 감수.

## 배경

이 앱은 `ts`를 처음부터 `2026-07-17T13:33:09.000+09:00` 형태로 저장해 왔고, 그 위에서 여러 기능이 조용히 같은 전제를 공유한다.

- **날짜/기간 자르기는 문자열 비교다.** `period.ts`가 하루·주·월 경계를 `+09:00` 문자열로 만들고, 달력·범위 조회가 `slice(0, 10)`로 날짜를 뽑아 사전순으로 비교한다. 이게 성립하는 건 모든 `ts`의 오프셋이 같기 때문이다 — 오프셋이 섞이면 `"...T23:00:00.000Z"`와 `"...T08:00:00.000+09:00"`의 사전순이 실제 시각 순서와 어긋난다.
- **시간 길이·간격은 절대 epoch다.** 체류 길이·이동 gap은 `Date.parse(end) - Date.parse(start)`로 계산한다. 이건 오프셋 표기가 달라도 안전하다. 즉 오프셋 통일이 진짜로 필요한 곳은 **경계 자르기** 쪽이다.

이 전제가 어디에도 "결정"으로 남아 있지 않으면, 신규 진입점이 다른 오프셋 문자열을 흘렸을 때 날짜가 하루 밀리는 버그가 조용히 생긴다(실제로 초기 import 경로가 그랬다). 그래서 결정으로 못박는다.

## 검토한 대안

### ① UTC(Z)로 저장하고 표시할 때 KST로 변환 — 기각

업계 표준(저장은 UTC, 표시는 로컬)이다.

**기각 이유: 이 앱의 핵심 연산이 "날짜 경계 자르기"인데 그게 매번 변환을 요구하게 된다.** 하루·주·월 통계, 달력 점, 일별 체류 목록이 전부 "KST 자정"을 경계로 삼는다. UTC 저장이면 이 경계를 만들 때마다 오프셋을 더해 재계산해야 하고, 문자열 사전순 비교라는 값싼 무기를 못 쓴다. 단일 타임존 앱에서 얻는 것 없이 복잡도만 는다.

### ② 기기 로컬 타임존 자동 추종 — 기각

저장 시각을 기기의 현재 타임존으로 직렬화한다.

**기각 이유: 저장된 문자열이 실행 환경에 의존하게 된다.** 여행 중이거나 기기 타임존이 바뀌면 같은 순간이 다른 오프셋 문자열로 저장돼, 과거 데이터의 경계가 흔들리고 사전순 비교가 깨진다. 재현 가능성(같은 입력 → 같은 저장값)이 사라진다.

### ③ +09:00 고정 — 채택

모든 진입점(수집기·import)에서 `ts`를 `+09:00` 고정으로 직렬화한다. import처럼 오프셋이 제각각인 외부 값은 `toKstIso(raw)`가 `Date.parse` → `getUTC*` 재조립으로 **실행 환경 타임존과 무관하게** +09:00으로 통일한다.

## 이 결정이 걸린 코드 (불변식 유지 지점)

| 위치 | 역할 | 불변식 의존 |
|---|---|---|
| `import/parseTimeline.ts` `toKstIso` | 외부 오프셋 → +09:00 정규화 | 모든 import ts를 통일 |
| `stats/period.ts` | 날짜/기간 경계를 +09:00 문자열로 생성, 정오 앵커 `Date`로 요일·월 계산 | 정오 앵커라 기기 TZ 무관 |
| `db/stays.ts` `getDatesWithData`·`getStaysByRange` | `slice(0, 10)` 날짜 사전순 비교 | 오프셋 동일 전제 |
| `stats/computeStats.ts` | 길이·gap은 `Date.parse` 절대 epoch, 날짜 버킷만 `slice(0, 10)` | 절대 시간은 오프셋 무관 |

## 결과

**얻는 것**

- 날짜·기간 경계 로직이 문자열 비교로 단순해진다(변환 레이어 불필요).
- 저장값이 실행 환경과 무관해 재현 가능하다.
- 타임존 지식이 진입점 정규화(`toKstIso`) 한 곳에 격리된다.

**잃는 것 / 지켜야 할 것**

- 다른 타임존 대응 없음 — 필요해지면 저장은 절대 epoch로 두고 표시 레이어에서만 TZ를 입히도록 후 변경한다.
- **모든 신규 진입점은 +09:00으로 정규화해야 한다** — 새 데이터 소스를 붙이면 저장 전 `toKstIso`(또는 동등 정규화)를 반드시 통과시킨다. 빠뜨리면 사전순 비교가 어긋나 날짜가 밀린다.
