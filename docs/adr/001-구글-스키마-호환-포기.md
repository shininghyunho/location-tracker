# ADR-001: 구글 타임라인 스키마 호환을 포기하고, 자체 최소 모델 + 손실 변환 import를 쓴다

- **상태**: 확정 (2026-07-17)
- **결정**: 앱은 자체 최소 모델(points → stays)로 동작한다. 구글맵 타임라인 내보내기 파일(timeline.json)은 import 어댑터 하나가 이 모델로 **손실 변환**한다. 구글 스키마와의 호환은 목표로 삼지 않는다.
- **대가**: import한 과거 기록은 구글맵 타임라인만큼 디테일하지 않다 — 장소명 자동 부여·이동수단 추정이 유실된다.

## 배경

이 앱은 스스로 백그라운드 GPS를 수집하는 것이 기본 동작이지만, 앱 설치 **이전**의 기록은 수집할 수 없다. 그 공백은 구글맵 타임라인의 수동 내보내기 파일(timeline.json)을 import해서 메꾸기로 했다(백필). 그렇다면 데이터 모델을 설계할 때 "구글 스키마와 얼마나 호환되게 만들 것인가"를 먼저 정해야 했다 — 수집 기능을 다 만든 뒤에 import가 안 맞으면 모델을 갈아엎어야 하기 때문이다.

실제 내보내기 파일(42MB)을 실측해 보니 구글의 구조는 2계층이었다.

- **rawSignals** — 원시 신호(좌표·활동 감지·와이파이 스캔). 단, 최근 30일 치만 남는다.
- **semanticSegments** — 구글이 원시 신호를 해석한 결과. "어느 가게에 방문(visit)", "무엇을 타고 이동(activity)" 같은 의미 정보가 붙어 있고, 전체 기간이 남는다.

## 검토한 대안

### ① 구글 스키마 100% 호환 — 기각

구글의 2계층 구조를 우리 모델로 그대로 채택하는 안. import가 무손실이 된다.

**기각 이유: semantic 정보는 우리가 재현할 수 없다.** visit의 장소 이름·집/직장 라벨은 구글의 장소 DB와 사용자의 구글 계정 데이터에서 나온다. 우리 앱이 직접 수집하는 데이터에는 이 재료가 없으므로, 스키마를 똑같이 만들어도 직접 수집분에서는 semantic 칸이 영원히 비어 있는 반쪽짜리 호환이 된다. 비어 있을 필드를 위해 복잡한 계층 구조를 유지하는 비용만 남는다.

### ② 원시 좌표만 저장 — 기각

points(시각+좌표)만 쌓는 가장 단순한 안.

**기각 이유: "장소" 개념이 없으면 앱의 목적을 달성할 수 없다.** 이 앱의 핵심 산출물은 "어디에 얼마나 있었는지" 통계인데, 좌표 나열만으로는 장소별 체류시간을 집계할 수 없다.

### ③ 자체 최소 모델 + 손실 변환 import — 채택

앱에 필요한 만큼만 모델을 정의한다: points(원시 좌표) → stays(체류, 사용자가 직접 라벨링). 구글의 의미 해석 중 재현 가능한 것(체류 판정)은 자체 규칙(stay-point detection)으로 직접 만들고, 재현 불가능한 것(장소명 자동 부여, 이동수단 추정)은 포기한다. timeline.json은 import 어댑터 하나가 이 모델로 변환하며, 변환 과정의 정보 손실을 허용한다.

## 결과

**얻는 것**

- 앱이 구글 데이터 없이 단독으로 완결 동작한다. import는 있으면 좋은 부가 기능일 뿐이다.
- 데이터 모델이 앱의 실제 필요만큼만 단순해진다.
- 구글 스키마 지식이 import 어댑터 한 곳에만 격리된다. 구글이 내보내기 포맷을 바꿔도 어댑터만 고치면 된다.

**잃는 것 (감수)**

- import한 과거 기록은 구글맵만큼 디테일하지 않다. 장소 이름은 자동으로 붙지 않고(사용자 수동 라벨), 이동수단 추정도 없다.
- 구글로의 역방향 export(우리 데이터 → timeline.json)는 목표에서 제외된다.
