# ADR-002: 웹뷰 래퍼는 Capacitor를 쓴다 (vs React Native, Flutter)

- **상태**: 확정 (2026-07-17) — 래퍼 결정은 유효. 백그라운드 위치 플러그인은 이후 Transistorsoft로 교체([ADR-004](004-수집엔진-Transistorsoft.md))
- **결정**: 크로스플랫폼 래퍼로 **Capacitor**를 쓴다. 웹(React)으로 만든 UI를 웹뷰로 감싸고, 백그라운드 GPS·SQLite만 네이티브 플러그인으로 쓴다.
- **대가**: 웹뷰 성능 한계(네이티브 UX 아님), 백그라운드 플러그인 품질 리스크 → PoC로 조기 검증.

## 배경

이 앱의 핵심 기능은 백그라운드 GPS 수집인데, 브라우저·PWA는 화면이 꺼진 상태의 위치 수집을 허용하지 않는다. 따라서 웹 기술만으로는 앱이 성립하지 않고, 네이티브 기능에 접근할 수 있는 크로스플랫폼 래퍼가 필요하다. 후보는 업계 표준 3개: Capacitor, React Native, Flutter.

전제가 되는 제품 원칙 두 가지:

- **웹 기반** — UI·로직은 웹 기술(React·DOM·CSS)로 만든다. 이 프로젝트는 웹 프런트엔드 훈련을 겸하므로, UI 개발이 곧 웹 개발이어야 한다.
- **최소 네이티브** — 네이티브가 꼭 필요한 기능은 백그라운드 GPS와 SQLite 딱 두 개다.

## 세 후보의 구조적 차이

세 후보는 "웹 기술을 얼마나 그대로 쓰는가"에서 갈린다.

```
Capacitor                React Native              Flutter
─────────                ────────────              ───────
React + HTML/CSS/JS      React(JSX) + JS           Dart
      │                        │                      │
   웹뷰가 그대로 렌더       JS는 로직만 담당,          자체 렌더링 엔진이
   (브라우저와 동일)       화면은 OS 네이티브          픽셀을 직접 그림
      │                   위젯으로 렌더                │
      ▼                        ▼                      ▼
  <div>, CSS 그대로      <View>, <Text> 전용        Widget 트리 전용
                         컴포넌트 (DOM 아님)        (웹 기술 아님)
```

| 기준 | Capacitor | React Native | Flutter |
|---|---|---|---|
| UI 작성 기술 | HTML/CSS/JS 100% 그대로 | React 문법이지만 DOM·CSS 아님 | Dart + Widget, 웹 기술 0% |
| 웹 기술 훈련 효과 | React·DOM·CSS·브라우저 전부 | React 지식만, DOM/CSS 없음 | 거의 없음 |
| 백그라운드 GPS | 플러그인 존재 | 플러그인 존재 | 플러그인 존재 |
| 성능·네이티브 UX | 웹뷰 한계 | 우수 | 가장 우수 |
| MVP 속도 | 웹 개발 그대로 | RN 학습 곡선 | Dart부터 새로 |

## 각 스택이 이기는 판

이번 기각이 "나쁜 스택이라서"가 아님을 기록해 둔다.

| 스택 | 이기는 상황 |
|---|---|
| Flutter | 앱 자체가 제품이고 성능·UX가 최우선일 때 |
| React Native | React에 익숙한 팀이 네이티브급 UX를 원할 때 |
| Capacitor | 웹 기술로 만들고 싶은데 네이티브 기능 몇 개만 필요할 때 ← **우리 상황** |

우리 앱은 UI가 웹으로 충분하고, 필요한 네이티브 기능이 두 개뿐이다.

## 결과

**얻는 것**

- "웹 기반" 원칙을 원칙 변경 없이 지킨다. RN·Flutter는 이 원칙 자체를 깨야 선택 가능했다.
- 웹뷰 방식만이 React+DOM+CSS+브라우저 동작 전부를 훈련시킨다.
- UI 개발이 곧 웹 개발이므로 MVP 속도가 가장 빠르다.

**잃는 것 / 리스크 (감수 방법 포함)**

- 네이티브 UX가 아니다 → 개인용 통계 앱이라 UX 완성도는 우선순위가 낮다.
- 웹뷰 성능 한계(대량 지도 마커 등) → MVP 데이터 규모에서는 문제되지 않는 것으로 판단, 문제 시 클러스터링으로 대응.
- 백그라운드 위치 플러그인이 커뮤니티 관리 품질이다 → PoC에서 검증한 결과 **커뮤니티판은 앱 종료·재부팅 생존 불가로 판명, Transistorsoft SDK로 교체해 해결**([ADR-004](004-수집엔진-Transistorsoft.md)). 리스크는 현실화됐지만 래퍼 결정 자체는 유효 — 웹뷰 래퍼 위에서 플러그인만 교체하면 됐다.
